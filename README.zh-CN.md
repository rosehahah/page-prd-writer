# Page PRD Writer

[中文](./README.zh-CN.md) | [English](./README.en.md)

一个面向产品团队的技能：把真实页面产物快速转成**简洁、可执行、可对齐**的页面需求文档。

`page-prd-writer` 不是通用型的 AI 文档生成器。它解决的是一个非常具体、但在日常协作里非常高频的问题：

**当页面方案已经大致成型后，如何把原型、截图、页面代码或路由信息，快速翻译成研发和运营都能直接对齐的 PRD。**

它默认不追求“写得很长”，而追求：

- 研发看得懂
- 运营能对齐
- 产品自己后续好继续补充
- 文档密度高，但不废话

## 这个技能为什么存在

产品日常工作里，真正耗时间的很多时候不是“想方案”，而是：

**方案其实已经有了，但还要再翻译成一份能给别人协作使用的需求文档。**

常见情况是：

1. 产品经理已经通过原型、页面草图、AI 出稿、现有页面参考，把页面结构想清楚了；
2. 功能方向和交互框架已经比较明确；
3. 但接下来还要把这些内容重新整理成一份能和研发、运营沟通的文档。

这个翻译动作，重复、费时，而且很容易写成冗长文档。

`page-prd-writer` 就是为这一步服务的。

## 它解决什么问题

很多 AI 写 PRD 的工具会走向三种常见问题：

- 套模板，写得很像 PRD，但不一定真能执行；
- 按页面可见区域逐块描述，信息很多，但重点不突出；
- 输出过长，像“完整”，但团队真正协作时并不好用。

这个 skill 的默认策略不同：

它会优先输出 **按功能归类、按规则表达、默认简洁** 的页面需求说明，只保留真正影响交付的逻辑，比如：

- 搜索行为
- Tab / 筛选逻辑
- 排序规则
- 列表/卡片操作
- 提交逻辑
- 空状态
- URL 参数同步
- 数据映射或输出规则

一句话概括：
**它写的是团队真正会拿来对齐的页面 PRD，不是为了“像 PRD”而写的长文档。**

## 适用场景

当你已经有比较具体的页面产物，希望快速转成需求文档时，这个 skill 最适合：

- 把原型页面转成页面 PRD
- 把前端已有页面反向整理成产品需求说明
- 把截图/页面录屏中的逻辑整理成页面需求
- 在和研发或运营开会前，快速沉淀一个对齐版本
- 在团队内部统一“简洁型页面 PRD”的写法

## 核心原则

### 1）先看真实页面产物，再写
它会优先读取真实输入，而不是凭空扩写。

输入可以是：
- 本地路由或页面路径
- 前端代码与相关常量
- 原型
- 截图

如果同时有 UI 和代码，默认以代码作为页面规则的更高优先级来源。

### 2）按功能分组，而不是按页面布局分块
输出会尽量按功能点来组织，而不是按“顶部区域 / 中间区域 / 按钮区域”这种视觉布局去写。

更接近实际协作讨论方式，比如：
- 列表排序
- 筛选逻辑
- 发布规则
- 卡片操作
- 结果状态

### 3）默认写短，不默认写全
默认目标不是“把页面所有东西都列出来”。

默认目标是：
**够执行、够对齐、够继续推进。**

一个普通页面，默认通常输出 3–8 个编号功能块。

### 4）只有在必要时才切详细版
只有在下面这些情况，才会切到更详细的输出：

- 用户明确要求“详细版 / 完整 PRD / 逐字段 / 全量需求”
- 页面本身是复杂发布表单 / 编辑表单

即便详细版，也依然强调紧凑，不写装饰性内容。

## 输出风格示例

```md
1、列表排序

排序选项包括：综合排序、质量优先、最近发布。默认选中综合排序。

综合排序：沿用搜索中台返回顺序。
质量优先：按推荐等级、推荐时间、创建时间倒序。
最近发布：按发布时间倒序，相同发布时间再按推荐等级倒序。
```

这就是它追求的密度：
一个功能点下面，只写真正影响执行的规则。

## 它会保留什么

- 排序规则
- 筛选项与组合逻辑
- Tab 切换逻辑
- 搜索行为
- URL 参数同步
- 提交 / 删除 / 复制 / 下载的结果影响
- 空状态
- 会影响输出的数据映射

## 它会主动删掉什么

- 装饰性的布局描述
- 重复的 UI 枚举
- 不影响逻辑的静态文案
- 过长的页面结构说明
- 为了“看起来完整”而堆出来的废话

## 对团队的价值

这个 skill 特别适合产品、研发、运营三方协作场景。

它的价值不是替代产品经理思考，而是：

**把“页面意图”翻译成“可交付文档”的成本压缩掉。**

这能让团队在下面这段链路里明显提速：

> 已经有页面方案 → 需要形成协作文档 → 进入研发/运营对齐

## 开源这个项目的出发点

这个仓库本质上是在做一件事：

**把真实工作流里高频、重复、但有判断门槛的一步，沉淀成可复用 skill。**

背后的判断很简单：

> 很多产品工作，不是从 0 到 1 想点子，而是把已经成型的方案，翻译成别人能执行的协作产物。

`page-prd-writer` 就是在把这一步结构化。
